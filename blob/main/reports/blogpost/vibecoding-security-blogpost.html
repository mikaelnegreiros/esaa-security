O experimento

Existe um termo que se popularizou na comunidade de desenvolvedores: vibe coding — a prática de descrever o que você quer, deixar o modelo gerar o código, aceitar o resultado e seguir em frente. Sem revisão rigorosa. Sem auditoria. Sem questionar o que o modelo decidiu omitir. Você confia na "vibe" e o sistema vai tomando forma.

Eu decidi testar os limites dessa abordagem de forma deliberada. Construí o ESAA-Supervisor — um sistema web completo para supervisão de agentes de IA, com backend em FastAPI, frontend em React/TypeScript, orquestração de múltiplos agentes CLI, streaming SSE de logs, gerenciamento de estado canônico baseado em filesystem, chat persistente e telemetria de tokens — usando exclusivamente modelos de IA para gerar o código.

O sistema funcionou. O roadmap canônico chegou a 18/18 tasks concluídas. A interface renderizava artefatos, o chat funcionava, os agentes executavam. Do ponto de vista funcional, era um sucesso que qualquer vibe coder celebraria.

Então eu fiz o que a maioria não faz: submeti tudo a uma auditoria de segurança completa, cobrindo 12 domínios, de autenticação a segurança de IA/LLM, seguindo referências da OWASP ASVS 5.0, CIS Controls v8.1 e NIST CSF 2.0.

O resultado: 19.5 / 100. Postura de segurança classificada como CRÍTICA. Não "precisando melhorar". Não "abaixo da média". Crítica — com vulnerabilidades que permitem execução remota de código, vazamento de credenciais e manipulação total de estado por atores não autorizados.

Note a ironia: os modelos geraram dependências limpas (92.3%) e frontend razoável (80%). Ou seja, as camadas que eles "veem" diretamente — bibliotecas populares, componentes React com padrões conhecidos — ficaram bem. Mas tudo que exige decisão arquitetural de segurança — autenticação, autorização, sanitização de prompts, controle de acesso ao filesystem — simplesmente não foi implementado.

O score global sofreu uma penalidade severa (Cap 50) pela presença de mais de 3 vulnerabilidades CRITICAL sem mitigação em domínios core. Isso reflete uma metodologia real: não importa quão bom é o frontend se a API está completamente aberta.

O padrão é claro

Modelos de IA otimizam para fazer funcionar. Quando você pede um endpoint que lista arquivos, o modelo gera um endpoint que lista arquivos — e funciona perfeitamente. Mas ele não se pergunta: "este endpoint deveria ser acessível sem autenticação?", "o caminho recebido deveria ser sanitizado?", "o CORS deveria ser restrito?".

Pior: quando geram adaptadores para CLIs de agentes, os modelos consistentemente escolheram os modos mais permissivos disponíveis. O Gemini foi configurado com --approval-mode yolo. O Claude Code com --permission-mode bypassPermissions. Não porque o sistema exigia, mas porque eram os caminhos de menor fricção para o código funcionar na primeira tentativa.

# O que os modelos geraram: command = ["gemini", "--approval-mode", "yolo", ...] command = ["claude", "--permission-mode", "bypassPermissions", ...] # O que deveria existir: command = ["gemini", "--approval-mode", "suggest", ...] # claude: sem flag de bypass, modo padrão com confirmação
O modelo não gera código inseguro por malícia. Ele gera código inseguro porque segurança é uma restrição que não aparece no prompt quando você está no modo "vibe".

Por que isso acontece

Depois de analisar os 3 relatórios completos da auditoria — resumo executivo, remediações técnicas detalhadas e boas práticas por domínio — três padrões emergem sobre como e por que os LLMs produzem código inseguro de forma sistemática:

Padrão 1

Otimização pelo caminho feliz. LLMs são treinados em uma distribuição massiva de código onde a maioria dos exemplos é funcional, não segura. Um app.add_middleware(CORSMiddleware, allow_origins=["*"]) aparece em milhares de tutoriais. É o caminho mais provável. O modelo converge para ele não porque é correto, mas porque é estatisticamente dominante.

Padrão 2

Ausência de contexto de ameaça. Quando você diz "crie um endpoint para listar arquivos de um diretório", o modelo não tem um modelo de ameaça. Ele não sabe que esse endpoint será exposto em rede. Ele não sabe que o input do usuário pode conter ../../etc/passwd. Ele produz exatamente o que foi pedido — e nada do que deveria ter sido inferido.

Padrão 3

Segurança como requisito não-funcional invisível. Autenticação, autorização, rate limiting, sanitização de logs, TTL de sessão, RBAC, CSP headers — nenhum desses aparece organicamente quando o prompt é orientado a funcionalidade. São requisitos transversais que exigem decisão arquitetural deliberada. E deliberação é exatamente o que o vibe coding elimina.

O paradoxo do vibe coding: Quanto mais fluida e produtiva a experiência de geração parece, mais invisíveis ficam as lacunas de segurança. A velocidade cria a ilusão de completude.

A remediação planejada exige 3 sprints totalizando ~61.5 horas de desenvolvimento. O Sprint 1 sozinho — focado em bloqueadores de produção como CORS, auth, sandboxing de paths e remoção dos modos bypass — demanda 14 horas. E isso para um sistema que "já funciona".

O relatório de boas práticas mapeou cada vulnerabilidade contra referências formais: OWASP Top 10:2021, OWASP ASVS 5.0, CIS Controls v8.1, NIST CSF 2.0 e NIST SP 800-53 Rev. 5. Não são opiniões. São padrões da indústria que existem justamente porque esses erros são recorrentes — e que nenhum modelo, em nenhum momento, citou ou seguiu proativamente.

O que isso significa para quem usa IA para codificar

Este experimento não é um argumento contra o uso de IA para gerar código. Os 5 modelos são extraordinários em produtividade, na qualidade do scaffold inicial, na resolução de problemas pontuais e na velocidade de prototipação. O argumento é contra a confiança cega.

Vibe coding funciona — para protótipos descartáveis. Para qualquer sistema que persista dados, aceite input externo, execute processos ou rode em rede, a revisão de segurança humana não é opcional. É o mínimo.

Recomendações práticas

1. Inclua segurança no prompt, não depois dele. Se o modelo não recebeu um modelo de ameaça, ele não vai gerar defesas. "Crie um endpoint que lista arquivos de um diretório, restrito ao diretório /srv/data, com autenticação JWT e validação de path traversal" produz um resultado radicalmente diferente de "crie um endpoint que lista arquivos".
2. Audite com o mesmo rigor com que você auditaria código humano. Código gerado por IA não merece um passe livre. Ele precisa de code review, checklist de segurança, SAST, secrets scanning e, idealmente, DAST em staging. O fato de vir de um modelo frontier não torna o output mais confiável — na verdade, a fluência do output pode tornar as falhas mais difíceis de detectar.
3. Trate os modos de execução do modelo como superfície de ataque. --approval-mode yolo, bypassPermissions, CORS: * — os modelos gravitam para a configuração de menor fricção. Cada flag de conveniência é uma decisão de segurança que precisa ser revista.
4. Segurança é arquitetura, não feature. Não dá para "adicionar segurança depois" a um sistema cujo design não previu autenticação, autorização ou isolamento. Os 61.5h de remediação do ESAA-Supervisor não são refatoração — são reconstrução de fundações. O custo de ignorar segurança no design inicial é sempre pago com juros.

O score de 19.5 não é sobre o modelo — é sobre o processo

Nenhum dos 5 modelos que usei é "ruim em segurança". Todos são capazes de gerar código seguro quando instruídos explicitamente para isso. O problema não está no modelo. Está no fluxo que elimina a fricção deliberada — aquela pausa onde um engenheiro pergunta "e se alguém enviar um path malicioso?", "e se o token não expirar?", "e se o CORS estiver aberto?".

Vibe coding otimiza para velocidade. Segurança exige deliberação. As duas coisas são fundamentalmente incompatíveis quando aplicadas sem supervisão.

O ESAA-Supervisor funciona perfeitamente. 18/18 tasks entregues. Interface responsiva. Agentes executando. Chat persistente. E, por trás de tudo isso, uma API completamente aberta, endpoints sem autenticação, filesystem navegável por qualquer ator na rede, prompts sem proteção contra injeção e agentes executando com permissões totais.

O código que funciona e o código que é seguro são dois conjuntos que se sobrepõem muito menos do que gostaríamos de admitir.

Se você está usando IA para gerar código — e deveria estar, porque a produtividade é real — lembre-se: o modelo entrega o que você pede. Segurança é o que você precisa pedir explicitamente.
